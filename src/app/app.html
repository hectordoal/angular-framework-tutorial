1. Justificación del Framework

Un Framework consiste en un conjunto estandarizado de herramientas y componentes que nos facilita una estructura base o inicial para comenzar a elaborar un proyecto. Podría decirse que es un punto de partida para la organización y desarrollo de un nuevo software.

A. ¿Para qué sirve?

Los frameworks tienen como finalidad abordar un proyecto en un menor tiempo, y en la programación, además, en un código más claro, más uniforme, ágil y también muy eficiente. El framework proporciona una estructura que los programadores pueden bien complementar o modificar en función de lo que necesitan. El uso de frameworks da como resultado, principalmente, el ahorro de los procesos de desarrollo porque podemos reutilizar módulos o herramientas.

B. Angular. ¿En qué consiste?

Angular es un framework de código de fuente abierta desarrollado por Google basado en TypeScript, orientado a la creación de aplicaciones web SPA (Single Page Application) rápidas, escalables y complejas gracias a su arquitectura basada en componentes que organiza el código en piezas reutilizables. Por tanto, es ideal para proyectos de frontend robustos al mismo tiempo que es compatible con una estructura MVC (Modelo-Vista-Controlador).

C. ¿Por qué Angular y no otro framework?

Porque optar por esta solución más que por otras alternativas:

  • Marco de aplicaciones completo y coherente: ecosistema con componentes, directivas, tuberías, formularios, router e inyección de dependencias.
  • Escala bien con fondos y código y la mantiene un equipo de Google.
  • TypeScript de primera clase y una CLI que estandariza el scaffold, las pruebas y los despliegues.
  • Enrutado oficial para SPA y navegación avanzada.
  • Formularios reactivos y basados en plantilla con validación nativa.
  • Soporte de SSR e hidratación para conseguir un mejor SEO y métricas de carga.
  • Ecosistema maduro y utilidades de testing listas para usar (Jasmine y Karma).
  • Arquitectura por componentes que hace fácil la mantenibilidad y las reutilizaciones a largo plazo.
  • Programación reactiva mediante Observables y señales cuando se necesita orquestar asincronía compleja.
  • Gran cantidad de documentación oficial y guías que cubren desde los fundamentos más básicos hasta la performance del framework.

D. Arquitectura de Angular

Angular es un marco modelo-vista-controlador (MVC) completo, proporciona una forma clara de cómo tiene que estructurarse la aplicación, además de proporcionar un flujo de datos bidireccional. Interactúa con el DOM de forma muy eficiente mediante su motor de detección de cambios, permitiendo una sincronización precisa entre la lógica y la interfaz.

E. Aplicaciones:

Es recomendable, por tanto, el desarrollo web que tenga como base Angular para aplicaciones empresariales de gran importancia, paneles de control internos, portales que tengan flujos de trabajo muy complejos y proyectos que requieran estándares de garantía, pruebas de integración y que sean escalables en el tiempo, así mismo, requerirá un mantenimiento predecible.

F. Lenguajes:

Se apoya principalmente en TypeScript (una versión mejorada de JavaScript), HTML y CSS.

G. Prerrequisitos:

Necesitas tener instalado un editor de texto como Visual Studio Code, Node.js instalado en su versión más reciente posible y conocer las bases de desarrollo web.

2. Instalación

A. ¿Se precisa instalación?

Si. A diferencia de una web más sencilla, Angular requiere de un entorno de ejecución llamado Node.js.

B. ¿Qué pasos hay que realizar para realizar la instalación?

Lo primero de todo es descargarnos el editor de texto, se recomienda Visual Studio Code. Lo podemos descargar en: [Visual Studio Code]

Una vez ya tengamos instalado Visual Studio, procedemos a descargarnos Node LTS en su versión más actualizada posible ya que así evitaremos problemas futuros relacionados con la versión. Lo podemos descargar en: [Node.js]

Despúes de descargarnos e instalarnos todo, abriremos Visual Studio y en el terminal ejecutaremos el comando npm install -g @angular/cli

Podremos verificar que lo tenemos todo listo y actualizado con el comando ng version

C. ¿Rutas para tener en cuenta?

Angular funciona bajo el concepto de SPA (Single Page Application). Esto significa que la aplicación nunca recarga el navegador por completo; en su lugar, utiliza una biblioteca oficial de enrutamiento para gestionar la navegación de forma interna. Este sistema se compone de tres pilares fundamentales:

- Las Rutas: Son objetos de configuración en el código TypeScript donde definimos la relación entre una URL (como /inicio) y el componente que debe mostrarse. Es el "mapa" de la aplicación.

- Los Outlets: Es una directiva especial (<router-outlet></router-outlet>) que actúa como un contenedor dinámico. Es el lugar exacto del HTML donde Angular "inyectará" el componente de la ruta activa.

- Los Enlaces: En lugar de usar el atributo href tradicional (que recargaría la página), usamos la directiva routerLink. Esto permite que el usuario cambie de vista instantáneamente sin perder el estado de la aplicación.

D. ¿Variables de entorno?

Las variables de entorno en Angular se gestionan mediante archivos específicos (normalmente en una carpeta llamada environments). Estas son fundamentales porque nos permiten definir configuraciones que cambian según dónde se ejecute la aplicación. Por ejemplo, podemos tener una variable para la dirección de un servidor de pruebas y otra para el servidor real definitivo; Angular se encarga de intercambiarlas automáticamente al compilar el proyecto.

F. ¿Necesitas base de datos?

Es importante aclarar que Angular, al ser una tecnología de Frontend (se ejecuta en el navegador del usuario), no contiene ni requiere una base de datos propia instalada. Para gestionar datos persistentes, Angular se conecta a un Backend (un servidor externo) a través de servicios que utilizan el protocolo HTTP. Si necesitamos guardar datos pequeños de forma local para que no se pierdan al cerrar el navegador, podemos utilizar herramientas como el LocalStorage o SessionStorage.

3. Primeros pasos

A. Cómo realizar una primera aproximación ("Hola mundo")

En Angular, todo se construye mediante llamadas a componentes. Cada componente consta de tres partes principales: el HTML (la estructura), el CSS (el diseño) y el TypeScript (la lógica). Para crear uno nuevo, usamos el comando en el terminal: ng generate component nombre-componente

Para mostrar datos dinámicos, Angular utiliza la interpolación. Esto nos permite "conectar" una variable que hayamos creado en nuestro archivo TypeScript (app.ts) directamente con nuestro HTML (app.html) usando las dobles llaves.

Para esta primera toma de contacto, hemos creado una variable en el archivo app.ts llamada HolaMundo que inicialmente está vacía. Al interactuar con la aplicación, haremos clic a un botón y le daremos un valor dinámicamente. En este caso un "Hola Mundo".

      export class AppComponent {
        HolaMundo: string = "";

        lanzarHolaMundo() {
          this.HolaMundo = "Hola Mundo";
          }
      }

B. Realizar la interacción

Para que el botón funcione, usamos el Event Binding de Angular escribiendo (click) seguido de nuestra función:

      <button (click)="lanzarHolaMundo()">
      Ejecutar Hola Mundo
      </button>

C. Ejecución del ejemplo

Al pulsar el siguiente botón, se ejecutará la función lanzarHolaMundo() de nuestro archivo TypeScript, asignando el texto a la variable y mostrándolo en pantalla mediante interpolación:

{{ HolaMundo }}

4. Utilización del Framework

A. ¿Cómo empezar a crear una aplicación base?

Para iniciar un proyecto desde cero, utilizamos el siguiente comando en el terminal: ng new nombre-de-tu-app. Al ejecutarlo, el CLI de Angular nos lanzará un asistente que nos preguntará si queremos añadir rutas o qué tipo de hojas de estilo (CSS, SASS) preferimos. Una vez finalizado, se genera automáticamente toda la estructura de carpetas, el archivo package.json con las librerías necesarias y un componente inicial listo para funcionar.

Una vez generado el proyecto, el trabajo principal se realiza dentro de la carpeta src/app, donde encontraremos el fichero app.html. Este es el archivo principal donde diseñaremos la interfaz de nuestra aplicación base.

B. ¿Cómo visualizar los resultados de la ejecución?

Para ver nuestra aplicación en funcionamiento o como la estamos modificando, seguimos estos pasos:

  1. Abrimos el terminal en la carpeta del proyecto y ejecutamos el comando: ng serve. En cambio, si queremos que abra automaticamente el navegador, utilizaremos el comando: ng serve -o
  2. Este comando compila la aplicación en memoria y activa un servidor local.
  3. Si lo abrimos automaticamente no hace falta, pero si no, hacemos control + clic en el enlace creado en el terminal, o de igual manera, abrimos nuestro navegador y escribimos la dirección: http://localhost:4200.

Una de las grandes ventajas es el Live Reloading. Si realizas un cambio en el código y guardas, la web se actualizará automáticamente en el navegador sin que tengas que refrescar manualmente.

5. Funcionamiento del Framework

A. ¿Actúa sobre páginas, ficheros o secciones?

Angular actúa sobre componentes dentro de una SPA (Single Page Application). No recarga "páginas" completas del servidor. En su lugar, modifica dinámicamente secciones del DOM (el árbol de la web) intercambiando componentes. El navegador solo carga un único fichero (index.html) y Angular se encarga de "dibujar" y "borrar" las secciones según la navegación.

B. ¿Cómo se editan o modifican los ficheros?

Los ficheros no se editan en el mismo servidor, sino en nuestro entorno de desarrollo (Visual Studio). El proceso funciona de la manera siguiente:

  • Modificamos el código TypeScript (lógica), HTML (estructura) o CSS (estilo).
  • El compilador de Angular detecta el cambio al guardar y realiza una Transpilación o Build.
  • Convierte todo ese código moderno en JavaScript plano para que el navegador pueda ejecutar al momento gracias al Hot Module Replacement.

C. ¿Cómo se referencian los ficheros desde otras aplicaciones?

El sistema de ficheros (componentes) de Angular es cerrado y modular, es decir, sus ficheros (componentes) no hace uso de rutas de ficheros al uso (../carpeta/archivo.js), sino que se hace mediante:

  • Selector: A partir de una etiqueta propia (ej. app-ejemplo) que podríamos usar en cualquier parte de la web.
  • Imports: En las versiones modernas, un componente "invoca" a otro importando la propia clase de aquel en el fichero TypeScript.
  • APIs: Si otra aplicación externa quiere "hablar" con nuestra app de Angular no va a acceder a sus ficheros, sino que se "habla" mediante servicios HTTP/REST o librerías exportadas como Web Components.
